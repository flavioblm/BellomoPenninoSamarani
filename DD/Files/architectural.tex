


\subsection{Overview: high-level components and interactions}
This section presents the overall system architecture by providing a high-level description of its components and the interaction between them. It outlines the architectural layers that composes the system and explains how they cooperate to deliver the required functionalities. Moreover, it discusses the key modeling and design choices adopted to satisfy the functional and non-functional requirements specified in the RASD document. \\[5pt]
The system’s architecture has been structured according to a four-tier model, a widely recognized design pattern that organizes the application into separate layers, each responsible for specific functions. This layered approach improves modularity, maintainability, scalability and fault tolerance, allowing each component to evolve independently without impacting the others.
The four tiers composing the architecture are the following:
\begin{itemize}
    \item \textbf{Presentation Tier:} Defines the user interface through which clients interact with the platform.
    \item \textbf{Routing Tier:} Contains all the components used to manage incoming requests, routing them to the appropriate services.
    \item \textbf{Application Tier:} Contains the core business logic and processes application data. Application Tier is designed according to a microservices-based architectural approach.
    \item \textbf{Data Tier:} Handles data storage, management, and access through databases.
\end{itemize}

\centering

\noindent{In particular, the Application Tier is designed according to a microservices-based architectural approach, where system functionalities are decomposed into independent and loosely coupled services. 
Although, for simplicity, the Application Tier and the Data Tier are represented in an abstract version as single components in the high-level architecture, in the actual implementation each microservice manages its own dedicated data store, in accordance with the database-per-service pattern. The internal structure and interactions of the individual services are described in detail in the following sections.}

\subsection{Component view}
The platform consists of several components grouped into two main categories: backend and frontend components. Notice that the backend comprehends Web Server and Application Logic tiers and it is organized according to a microservices architecture. Moreover, each component of the software is responsible for providing a specific service and implements one or more system functionalities.\\[8pt]
\textbf{Backend components}
\begin{itemize}
    \item \textit{Authentication microservice}
    \\ This service lets the users to perform sign up and login in the application. If a new user sign up, the service checks the email does not exist in the system or if an user is logging the system checks that the credentials are corrects.
    \item \textit{Account microservice}
    \\ This service stores and updates the cyclist profile information. It permits the cyclist edit his username, email, password and photos.
    \item \textit{Inventory microservice}
    \\ This service saved and shows cyclist's personal routes and paths. It supports filters by date, distance and score . When the cyclist open a path it returns the list of all the routes of that path and details
    \item \textit{ManagePath microservice}
    \\This service manages paths in the BBP. It creates automatically a new path when it is created a route on a non existing path. It manages the visibility of the path (public or private) and eventually the deletion of the path.
    \item \textit{ManageRoute microservice}
    \\This service manages routes, a single ride on a path. It lets user create a route either in manual mode or automatic mode. It also enriches the route with other details and supports the operation of publish, unpublish and delete.
    \item \textit{Navigation system microservice} 
    \\ This service implements a navigation support for the cyclist. When a cyclist selects a path the navigation system guides the cyclist along the streets of that path until the end point is reached.
    \item \textit{Recommendation system microservice} 
    \\ This service supports the user search of a path. From a start and an end point received in input it recommends a list of path ranked by score. Moreover, the service merges different information from several routes on the same path keeping the path status updated.
    \item \textit{ManageSensorsData microservice}
    \\This service collects data from the mobile sensors (GPS,accelerometer,gyroscope) during the ride of the cyclist. It enriches the route with these information.
    \item \textit{SensorData}
    \\This is a subcomponent of the \textit{User Mobile App} component and cooperates with \textit{ManageSensorsData microservice}, periodically sending data about device sensors.
    \item \textit{API Gateway}
    \\ This is the entry point for each app request. It forwards each request to the right microservices.    
    \item \textit{DBMSs}
    \\Each microservice owns its own database to store its data, keeping the services independent.
\end{itemize}
\textbf{Frontend components}
\begin{itemize}
    \item \textit{User Interface}
    \\This is a subcomponent of the \textit{User Mobile App} component and represents the app interface for users. It allows accessing all the features of BBP for the Cyclist and a smaller portion of them for the Guest. 
\end{itemize}
\textbf{External components}
\begin{itemize}
    \item \textit{WeatherCondition API}
    \\This external service provides weather data and is called by the \textit{ManageRoute service} and to enrich the details of a route with this piece of data.
\end{itemize}

Additionally, it is useful to provide a more in-depth description of a generic microservice component. It is based on the \textbf{Controller-Service-Repository pattern}, which adds modularity and promotes separation of concerns within the component. The sub-components of a microservice operate as follow:
\begin{itemize}
\setlength{\itemsep}{0pt}
    \item The \textit{Controller} is responsible for exposing the component functionalities and it is based on the RESTful architectural style.
    \item The \textit{Service} acts as intermediary between the two other components of the microservice and contains all the business logic.  
    \item The \textit{Repository} is responsible for querying the DBMS to retrieve and persist data.
    \item The \textit{Model} is used to map a database table into an object.
    \item The \textit{DTO} is the object used to transfer data during RESTful request/response. 
\end{itemize}

\subsection{Deployment view}
\begin{figure}[H]
\centering
\includegraphics[width=1\textwidth]{Images/DeploymentViewDiagram.png}
\caption{Deployment View Diagram}
\label{fig:DeploymentView}
\end{figure}

This is the full deployment view of the system.

\noindent The client side is a \textbf{Mobile Device}. Inside the device there is a mobile operating system (Android or iOS). The \textbf{BBP App} is installed on the OS and communicates with the backend to obtain the required application data. In addition, the BBP App reads sensor data (GPS, accelerometer, gyroscope) through local operating system APIs. The device sends external requests using the \textbf{HTTPS} protocol.

\noindent All requests from the mobile device first reach the \textbf{API Firewall}. The purpose of this firewall is to filter incoming traffic and block malicious or unauthorized requests. After the firewall, traffic enters the internal network and, from this point, it is forwarded using \textbf{HTTP}.

\noindent The backend is deployed on three replicated \textbf{Application Servers} to ensure high availability and fault tolerance. Each application server is associated with an \textbf{API Gateway}. A \textbf{Load Balancer} distributes incoming requests across the three replicated servers.

\noindent Each application server hosts the business logic of the system. Every microservice is deployed in its own container to keep services isolated, with independent dependencies and configuration. This improves portability, simplifies deployments, and enables better resource control.

\noindent All application servers communicate with the database layer to store and retrieve data using the \textbf{TCP} protocol. The database is replicated on three servers, so data remains available even if one DB server has issues, and duplication increases reliability. The three database servers are synchronized through software replication mechanisms to keep data consistent across replicas. The \textbf{Database Firewall} protects the database layer by filtering and controlling incoming connections from the application servers, reducing the risk of attacks and data leaks.

\noindent The database layer is also organized using containers. Each container groups the schemas required by a specific microservice, so every service has a dedicated database structure and data remains isolated from the others.

\subsection{Selected Architectural Styles and Patterns}
\textbf{Four-tier client-server architecture}\\ 
The system is designed using a four-tier client-server architecture, meaning that the solution is divided into four logical layers, each one with a specific responsibility.
This gives the system a clear structure, every tier focuses on specific types of task and tiers separation makes the architecture easy to understand, maintain and evolve over time.
Each tier can be scaled or updated independently without impacting the others.
\\[5pt]
The first tier is the \textbf{Presentation tier}, which is introduced to separate user interface concerns from business logic. This architectural choice enables independent evolution of the frontend components without affecting backend services.
It contains a single client, the BBP APP installed on a mobile device. This tier is responsible for the user interaction through the User Interface and for collecting device data such as the sensor data. The client tier interacts with the application sending requests to the backend and waiting for the responses.
\\[5pt]
The second tier is the \textbf{Routing tier} which includes all the components that control how requests enter the system and how they are routed inside the architecture. This tier is implemented with elements like the Load Balancer, API Gateway and the Proxy Server. \\
The \textit{Load Balancer} is important because it distributes requests across multiple APIs Gateway so the system can handle high traffic and continue working even if one server becomes unavailable. \\
The \textit{API Gateway} is a controlled entry point to the backend services, so this part is necessary because it routes each request to the correct microservice. \\
The \textit{Proxy Server} is used to support routing and to implement a caching mechanism. The aim of this tier is to improve both security and scalability centralizing the management of incoming traffic. 
\\[5pt]
The third tier is the \textbf{Application Server tier}. It is where the business logic of the application is executed. The application tier is replicated across multiple Application Servers to guarantee high availability and fault tolerance. Each server hosts the same set of microservices and every microservice is deployed in its own container. In this way each microservice can be updated, restarted and scaled independently. This tier communicates internally using HTTP protocol and satisfy all the requests from the API Gateway.
\\[5pt]
The fourth tier is the \textbf{Data tier}, which contains the persistent storage of the system. The decision to separate this tier from the application tier depends on protecting the data and keeping the storage concerns independent from the business logic. The databases are deployed with replication across multiple servers, so data remains available even if one database server has problems. In addition, there is a software mechanism that synchronize the servers to keep the information consistent. Each microservice has a dedicated DBMS to interact for retrieving and saving data, according to the \textit{database-per-service} pattern. 
\\[5pt]
The four-tier architecture is chosen because it enforces separation of concerns and produces a modular system in which each layer has a well-defined role. This structure helps the systems to be more scalable because the different tiers can be replicated and scaled horizontally if the app dimensions or the amount of the requests grow up. In addition the architecture makes the system more secure, more reliable and easier to build.
\\[8pt]
\textbf{Microservice architecture}\\
The decision to implement a microservices architecture stems from various factors related to both the product and the development process. \\[5pt]
From a product perspective, microservices enable fine-grained scaling through flexible deployment and selective replication of individual services, which is not possible in monolithic systems. This architectural choice also improves resilience by limiting the impact of localized failures, allowing the system to continue operating even in the presence of partial service outages.
Additionally individual services can be reused across different contexts and combined to support multiple workflows.\\[5pt]
From a process perspective, this architecture reduces team synchronization overhead by enabling small, autonomous teams with well defined responsibilities. 
\\[8pt]
\textbf{Security pattern}\\[5pt]
The security is granted thanks to the API Gateway. It represents a controlled entry point to the backend services and is responsible for managing all incoming requests before they reach the application tier, forwarding them to the appropriate microservices. This architectural choice prevents direct access to internal services and improves system security by centralizing the management of incoming traffic and access control mechanisms.\\[3pt]
The API Gateway also contributes to system scalability and reliability. To avoid having a single point of failure, multiple API Gateways instances are deployed and a Load Balancer is used to distribute requests among them. This ensures high availability and allows the system to handle high traffic while maintaining correct operation even in the presence of component failures.
\\[8pt]
\textbf{Resiliency pattern, circuit breaker}\\
In order to avoid delays and overload when a microservice is failing there is a circuit breaker at the entry point of each microservice and also on the existing communication links between microservices. Every time a request is sent to a microservice, the circuit breaker monitors the outcome of the call. If the call succeeds, traffic continues normally. If the call fails or the response time becomes too high, the circuit breaker switches to an open state and stops forwarding requests, immediately returning an error or fallback response instead of waiting.
\\The circuit breaker prevents components from continuously invoking a microservice that is slow or down reducing useless resource consumption. As a result, the architecture remains more stable and resilient and healthy microservice can keep working even when another service is temporarily unavailable.
\\[8pt]
\textbf{Communication pattern}\\
In the system both synchronous and asynchronous communication are used. The communication pattern is chosen depending on the nature and the expecting duration of the interaction. \\[5pt]
The interactions with the user are handled \textit{synchronously}, as they require immediate feedback to ensure a responsive user experience. In synchronous communication, both the client and the service are actively involved at the same time, allowing requests to be processed and responses to be returned without delay. \\[5pt]
Otherwise, operations that take more time or do not need an immediate response are handled asynchronously, such as data saving and interactions with external services and device sensors. Asynchronous communication enables the system to decouple the involved components, allowing each participant to proceed independently and improving system scalability and resource use. \\[8pt]
\textbf{Database-per-service pattern}\\
The Database-per-service pattern means that each microservice owns and manages its own dedicated schema instead of using a single shared database for the whole system . This choice is consistent with a microservices architecture because it keeps services independent, every microservice can design its data model in the best way for its domain and can evolve its schema overt time without forcing changes on other services . Another benefit is that there is no single logical point of failure at the data level, the separation also improves security because access rules can be applied per service and each microservice can be allowed to access only its own data.
When different microservices share information, their database are synchronized at the application lever, this approach preserves service autonomy and leads to eventual consistency, where data becomes consistent across services after the synchronization messages are processed.
\subsection{Other design decisions}
\textbf{Firewalls} \\
The use of firewall is a ley security decision in the architecture. There are two firewalls that control and filter traffic in two critical points: the entrance of the routing tier and the entrance of the database layer.
The \textbf{API Firewall } is placed at the boundary between the external world and the internal infrastructure. Its main role is to protect the backend APIs from malicious traffic coming from the internet. It inspects incoming request, applies security rules, filters attacks and abnormal request patterns. 
In practice, it ensures that only valid and safe calls can enter the internal network providing a first level of defense before traffic reaches the routing system and the application servers.
\\
\\
\textbf{Proxy}\\
The system includes a \textbf{proxy server} mainly configured to provide a caching mechanism. This means that when the same information is requested multiple times, the proxy can store the response and serve it again without forwarding every request to the backend.
Thanks to this approach, the load balancers send requests to the proxy servers instead of querying the API Gateways directly. If the proxy already has a valid cached response, it can return the data immediately. This reduces the load on the API gateways and application servers and improves response time.
\\
\\
\textbf{Duplicated Servers and Load Balancer}\\
The deployment has multiple replicated application servers and a load balancer to guarantee high availability,scalability and fault tolerance. 
The \textbf{load balancer}  acts as a single entry point for backend traffic and distributes incoming requests across the application servers. This prevents one server from becoming a  bottleneck and allows the system to handle higher workloads more efficiently. 
Having \textbf{multiple application server} also ensure that the service remains available even if one server fails or becomes overloaded.
